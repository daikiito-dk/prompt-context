ルーツと時系列 — どこから枝分かれしたか

大規模言語モデルの実用化を土台に、補完 → 対話 → エージェントへと能力が伝播してきた流れを、樹形と年表で示します（各項は代表的な傾向であり、厳密な年代認定ではありません）。

ルーツ Transformer 以降の LLM · API 化 · OSS ウェイト
モデル API クラウド推論・トークン課金・マルチモーダル拡張
IDE 統合 LSP 周辺 · 拡張機能 · ワークスペース文脈
エージェント層 ツール呼出 · ターミナル · MCP・オーケストレーション
  1. 〜2022
    インライン補完・ペアプロ型

    エディタ内での行単位提案。GitHub Copilot などが一般化。

  2. 2023 前後
    チャット UI の横行

    IDE サイドバー・Web の対話からリファクタや説明を依頼する運用が定着。

  3. 2024〜
    マルチステップ・自律フロー

    リポジトリ横断編集、CLI エージェント、MCP によるツール接続が製品化。

目的別 — 何を達成したいかで選ぶ（4 章）

同一のモデルでもプロダクト形態が異なります。下の 4 タブは「よくある検討軸」の切り口です。

実装スピードとタスクの完遂

  • Cursor — AI ネイティブ IDE、ルール・エージェント・MCP
  • Windsurf — Flow / Cascade による広いコンテキスト編集
  • GitHub Copilot — 主要 IDE への統合、チャット・エージェント機能
  • Claude Code — CLI 起点のリポジトリ作業（Anthropic 系）

可読性・安全側面のレビュー

  • IDE 内チャットによる差分説明・エッジケース洗い出し
  • 静的解析・SAST と併用したPR コメント自動化（各種クラウドサービス）
  • チームルールを Rules / AGENTS.md 等でモデルに渡す運用

パイプラインと環境の再現性

  • CI/CD（GitHub Actions 等）でのテスト・Lint 自動化との組み合わせ
  • Dev Containers / 環境 as Code で「同じ文脈」を共有
  • リリースノート・変更履歴のドラフト生成に AI を利用する事例

ポリシー・監査・データ境界

  • 送信データの範囲（学習利用の可否・保持期間）の契約確認
  • 組織 SSO・リポジトリ単位の許可・ログ保全
  • 下の「導入・評価の観点」とあわせて社内チェックリスト化

基盤モデル別 — どの API／家族かで見る

フロントプロダクトは複数モデルを切り替える場合もあるため、契約・レイテンシ・コンテキスト長の議論では「ベンダー／モデル系譜」が整理しやすいです。

OpenAI 系

GPT ファミリ、推論 API、エージェント向けツール呼び出し。

  • ChatGPT / API / Agentic 機能
  • 多数のサードパーティが OpenAI API を既定バックエンドに採用

Anthropic 系

長文脈・指示追従の議論で参照されやすい Claude 系。

  • Claude（Web / API）
  • Claude Code（CLI）

Google 系

Gemini・Vertex 経由・Android Studio 等の開発者向け統合。

  • Gemini
  • Google AI Studio

その他・オープンウエイト

オンプレ／VPC 内ホスティングや、複数バックエンドの抽象化レイヤ。

  • Mistral、Llama ファミリ等 — 推論ホスティングはクラウド・自前 Build 双方
  • ルーター・ゲートウェイ製品でモデルを動的スイッチ

開発環境 — エディタとターミナル

AI 機能は拡張機能・公式統合・CLIとしてここに接続されます。既存標準を変えずに載せるか、専用 IDE に移るかが主な二分法です。

Visual Studio Code

最も普及した拡張モデル。Copilot・各種チャット拡張・ローカルモデル連携の実例が多い。

JetBrains IDE

IntelliJ / PyCharm / WebStorm など。組み込み AI アシスタントやプラグインで統合。

Visual Studio

.NET 中心の大規模ソリューション。マイクロソフト製エコシステムと連動。

ターミナル / CLI

シェルスクリプト、Make、コンテナ内での エージェント CLI 実行。CI との親和性が高い。

データ層 — アプリと RAG・状態管理

コード生成だけでなく、スキーマ・マイグレーション・ベクトル検索周りのツール選定が一体となります。

PostgreSQL · MySQL

リレーショナル中心の業務データ。マイグレーション（Flyway / Alembic / Prisma 等）とのセットが一般的。

Redis · キャッシュ

セッション・レート制限・キュー。AI 機能の後段に載せることが多い。

ベクトルストア

埋め込みによる RAG。Pinecone / Weaviate / pgvector 等、運用形態はクラウドからセルフホストまで幅あり。

Supabase など BaaS

Auth・DB・Edge を束ね、プロトタイプから本番まで短時間で立ち上げる構成。

P&C が想定する運用の枠組み

ツールが変わっても再現しやすいよう、Contexts とプロンプト資産で前提を固定する考え方です。

変更内容の共有フォーマット

差分、意図、既知のリスクを決まった書式で記載。

評価用データのスナップショット

入力と期待挙動をバージョン管理し、プロンプト改修の影響を追跡。

ガバナンス用メタデータ

機密度・PII・保管期間をタグ化し、利用境界を明文化。

導入・評価の際の観点

AI 支援は情報の取り扱い・契約・監査に直結します。

  • データ取り扱い — 送信・学習・保持の条件を利用規約で確認。
  • 認証・権限 — SSO、リポジトリ境界、アカウントの連係。
  • 利用ログ — 説明責任のためのトレーサビリティ。
  • モデル更新 — ベンダーポリシー変更の定期確認。

一般的な整理であり、法的助言ではありません。ご組織の専門部署にご相談ください。
